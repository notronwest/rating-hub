Review points — unified model

Thesis: sequences, flags, and faults are really the same thing — a shot worth reviewing, with optional context and optional coach metadata. The difference between them is just variation on two axes:
  • Origin — did a coach flag it (⚑), or did the system auto-detect a fault (●)?
  • Size — is it a single shot, or a context window of shots around it?
So one list, one row, one edit panel. Origin is a small glyph, size is a small chip. Everything else is the coach's work on top.

Unified detail / edit panel

Same form regardless of origin. Context size is a first-class control (±shots) so a 1-shot flag can grow into a sequence, and a fault can pull in setup shots.

8review points 3 reviewed · 5 untouched
0:45 ▶ smash Celeste · rally 4 · shot 7 ⚑ coach-flagged
Before: After: Currently: 1 shot

Same panel handles: promoting a flag to a sequence (bump "after" to +3), adding diagnosis to an auto-detected fault (fill FPTM), or dismissing an irrelevant one. Nothing is a separate kind — just different data on the same object.

Model summary

One table under the hood, probably review_points, with:
  • anchor_shot_id — always single
  • origin — "coach" | "auto_fault"
  • context_before, context_after — window size (0/0 for a bare flag, 3/3 for a typical sequence)
  • note, fptm, drill, dismissed — coach metadata
Existing tables (analysis_flagged_shots, game_analysis_sequences, and the dismissed_loss_keys concept) can stay — we just render them all through one lens. Or we can migrate to a unified table later if it feels cleaner.
